# iOS ‚Äî MASVS-CRYPTO (static)

Static checks for cryptographic strength/usage and key management in an iOS artifact (`.ipa` / `.app` bundle / Mach-O), via CommonCrypto / CryptoKit / Security.framework / Keychain and bundled crypto libs.

Anchor each finding to the exact control in SKILL.md (read it); verify MASWE/MASTG IDs against the live guide.

> **FairPlay caveat.** Against an App Store binary (`cryptid=1`), `otool`/`nm`/`class-dump`/Ghidra read **ciphertext** for `__TEXT` ‚Äî every crypto string/symbol/control-flow conclusion is **invalid until the binary is decrypted** (see the Tools/FairPlay section of the iOS static parent file). A static-only crypto review of an encrypted `.ipa` is fundamentally limited; decrypt first, then re-run these checks. A development/ad-hoc/enterprise build is usually already decrypted (`cryptid=0` via `rabin2 -I`).

## Group controls (MASVS v2.0.0, verbatim)
- **MASVS-CRYPTO-1** ‚Äî *The app employs current strong cryptography and uses it according to industry best practices.* (weak/broken algos & modes, ECB, static/reused IVs, MD5/SHA-1 for security, missing/weak MAC, bad RNG)
- **MASVS-CRYPTO-2** ‚Äî *The app performs key management according to industry best practices.* (hardcoded keys/IVs/salts, weak key generation/derivation, key not protected at rest / access not restricted, Secure Enclave not used where expected, key reuse across purposes)

Everything below is a **candidate generator** ‚Äî promote only after the anti-false-positive gate in SKILL.md. Pick the control **number** by statement match (the line above), not by group alone. Live test URL: `https://mas.owasp.org/MASTG/tests/ios/MASVS-CRYPTO/<ID>/`. The MASTG is mid-refactor: the legacy umbrella iOS CRYPTO tests `MASTG-TEST-0061` (algorithm config), `-0062` (key management), `-0063` (RNG) are still live but are being split into atomic, MASWE-mapped tests (`-0209`/`-0210`/`-0211`/`-0213`/`-0214`/`-0311`/`-0317`/`-0349`); **prefer the atomic one and note the legacy umbrella.** The iOS CRYPTO atomic test set was confirmed against the live `tests-beta/ios/MASVS-CRYPTO` catalog (see notes).

## Checks

### CRYPTO-1 ‚Äî strong crypto, used correctly
- **Weak/broken symmetric primitives** ‚Äî DES/3DES/RC4/Blowfish via **CommonCrypto** (`CCCrypt`, `kCCAlgorithmDES`/`-3DES`/`-RC4`) or a bundled lib (OpenSSL/CryptoSwift). ‚Üí **MASVS-CRYPTO-1**, **MASWE-0020** (Improper Encryption), **MASTG-TEST-0210** (Broken Symmetric Encryption Algorithms, atomic); legacy umbrella `MASTG-TEST-0061`.
- **Broken hashing for security** ‚Äî MD5/SHA-1 (`CC_MD5`, `CC_SHA1`, CryptoKit `Insecure.MD5`/`Insecure.SHA1`) used for signatures / password hashing / integrity of trusted data. ‚Üí **MASVS-CRYPTO-1**, **MASWE-0021** (Improper Hashing), **MASTG-TEST-0211** (Broken Hashing Algorithms, atomic); legacy `-0061`. *(Checksum/cache-key use is a classic FP ‚Äî see below.)*
- **Insecure modes / padding** ‚Äî ECB (`kCCOptionECBMode`); CBC with no integrity (see MAC bullet). ‚Üí **MASVS-CRYPTO-1**, **MASWE-0020**, **MASTG-TEST-0317** (Broken Symmetric Encryption Modes, atomic); legacy `-0061`. Risky padding (e.g. PKCS#1 v1.5 where OAEP expected) ‚Üí **MASWE-0023** (Risky Padding), iOS atomic ID to confirm.
- **Predictable / reused IV or nonce** ‚Äî constant or zero IV passed to `CCCrypt` (static-IV CBC), or **GCM/ChaChaPoly nonce reuse** in CryptoKit (`AES.GCM.seal(_:using:nonce:)` / `ChaChaPoly.seal` with a fixed/counter nonce ‚Äî catastrophic, leaks the auth key). Grep for hardcoded IV byte arrays and reused `AES.GCM.Nonce`. ‚Üí **MASVS-CRYPTO-1**, **MASWE-0022** (Predictable Initialization Vectors), iOS atomic ID to confirm ‚Äî cite control + "Predictable/Reused IV or Nonce (ID to confirm)"; rel. `MASTG-TEST-0317`.
- **Missing / improper MAC (unauthenticated encryption)** ‚Äî CBC/CTR without a separate MAC, MD5/SHA-1-based HMAC, CRC32 used as integrity, or truncated tags on data that is trusted after decryption. Prefer AEAD (`AES.GCM`/`ChaChaPoly`). ‚Üí **MASVS-CRYPTO-1**, **MASWE-0024** (Improper Use of MAC), iOS atomic ID to confirm ‚Äî cite control + "Improper/Missing MAC (ID to confirm)".
- **Bad RNG for security material** ‚Äî keys/IVs/nonces/tokens/salts from non-CSPRNG: `rand()`/`random()`/`arc4random()`/`arc4random_uniform()` or Swift `Int.random`/`UInt8.random` misused for secrets, instead of `SecRandomCopyBytes` or CryptoKit's `SymmetricKey(size:)`. ‚Üí **MASVS-CRYPTO-1**, **MASWE-0027** (Improper Random Number Generation), **MASTG-TEST-0311** (Insecure Random API Usage, static atomic; runtime counterpart `MASTG-TEST-0349`); legacy umbrella `MASTG-TEST-0063`.
- **Risky / non-standard crypto implementation** ‚Äî home-rolled ciphers, XOR "encryption", or an unvetted third-party crypto lib instead of platform CommonCrypto/CryptoKit. ‚Üí **MASVS-CRYPTO-1**, **MASWE-0019** (Risky Cryptography Implementations), iOS atomic ID to confirm.

### CRYPTO-2 ‚Äî key management
- **Hardcoded keys / IVs / salts in code** ‚Äî literal `Data`/`[UInt8]`/`String` passed to `SecKeyCreate*`, `SymmetricKey(data:)`, `CCCrypt`, or `IvParameterSpec`-equivalents; AES/HMAC keys as string constants. ‚Üí **MASVS-CRYPTO-2**, **MASWE-0014** (Cryptographic Keys Not Properly Protected at Rest), **MASTG-TEST-0213** (Use of Hardcoded Cryptographic Keys in Code, atomic); legacy umbrella `MASTG-TEST-0062`. *(MASWE-0013 was merged into 0014 ‚Äî use 0014.)*
- **Hardcoded keys / IVs / salts in resources** ‚Äî key material in bundled files (`.plist`, `.json`, `.pem`/PKCS, embedded DBs, assets). ‚Üí **MASVS-CRYPTO-2**, **MASWE-0014**, **MASTG-TEST-0214** (Hardcoded Cryptographic Keys in Files, atomic); legacy `-0062`.
- **Insufficient key sizes** ‚Äî RSA/DSA/DH < 2048-bit, EC < 256-bit, symmetric < 128-bit (inspect `SecKeyCreateRandomKey` attrs / `kSecAttrKeySizeInBits`, CryptoKit curve choice). ‚Üí **MASVS-CRYPTO-2**, **MASWE-0009** (Improper Cryptographic Key Generation), **MASTG-TEST-0209** (Insufficient Key Sizes, atomic); legacy `-0061`/`-0062`.
- **Weak key derivation (KDF)** ‚Äî `CCKeyDerivationPBKDF` with low iterations / missing or static salt / short derived key, `CC_SHA256(passphrase)` used directly as a key instead of a KDF, or weak HKDF/scrypt params. ‚Üí **MASVS-CRYPTO-2**, **MASWE-0010** (Improper Cryptographic Key Derivation), iOS atomic ID to confirm ‚Äî cite control + "Weak Key Derivation (ID to confirm)"; legacy `-0061`/`-0062`.
- **Keychain accessibility too weak / key not protected at rest** ‚Äî keys (or key-equivalent secrets) added with `kSecAttrAccessibleAlways`/`...AlwaysThisDeviceOnly` (deprecated, always readable) or `...AfterFirstUnlock(ThisDeviceOnly)` for high-value keys, or **omitting `*ThisDeviceOnly*`** so the item rides iCloud Keychain backup to attacker hardware. Baseline for sensitive keys: `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`. Grep the `SecItemAdd`/`SecAccessControlCreateWithFlags` accessibility argument. ‚Üí **MASVS-CRYPTO-2**, **MASWE-0018** (Cryptographic Keys Access Not Restricted; rel. **MASWE-0014** at-rest), iOS atomic ID to confirm ‚Äî cite control + "Keychain Accessibility Class (ID to confirm)"; legacy umbrella `MASTG-TEST-0062`. *(The related storage-API test `MASTG-TEST-0300` is anchored under MASVS-STORAGE, not CRYPTO ‚Äî don't mis-cite it here.)*
- **Secure Enclave not used where expected (L2/R)** ‚Äî software-only keys where hardware-backed keys are warranted: absence of `kSecAttrTokenIDSecureEnclave` on `SecKeyCreateRandomKey`/`SecAccessControlCreateWithFlags`, or CryptoKit `SecureEnclave.P256.*` not used for the signing/key-agreement key. Profile-dependent (expected at L2/R for high-value secrets). ‚Üí **MASVS-CRYPTO-2**, **MASWE-0014** (rel. **MASWE-0018**), Secure-Enclave-specific iOS atomic ID to confirm; legacy `-0062`.
- **Key reuse across purposes** ‚Äî same key used for both encryption and MAC, or sign and encrypt (violates NIST SP 800-57 key separation). One bullet ‚Äî low static frequency. ‚Üí **MASVS-CRYPTO-2**, **MASWE-0012** (Insecure or Wrong Usage of Cryptographic Key), iOS atomic ID to confirm; legacy `-0062`.

> **Note on compiler hardening / RNG-at-rest:** PIE/stack-canary/ARC are **MASVS-CODE**, not CRYPTO (`MASTG-TEST-0228`/`-0229`/`-0230`, MASWE-0116) ‚Äî assess them in the CODE reference. NSFileProtection, plaintext stores, and Keychain *storage choice* for non-key secrets are **MASVS-STORAGE** (`MASTG-TEST-0300`). Biometric binding of a key's release (`SecAccessControlCreateWithFlags` + LAContext) is **MASVS-AUTH-2** ‚Äî analyze it there. Keep this file to crypto strength/usage and key management.

## Classic false positives
Consult before promoting a candidate. Each is **usually NOT a finding** absent extra evidence:

| Candidate | Why it's usually a false positive | When it *is* real |
|---|---|---|
| MD5 / SHA-1 in CommonCrypto or `Insecure.*` | Used for a non-security checksum, ETag, or cache key | used for signatures, password hashing, or integrity of trusted data |
| Strings/symbols "leaking" key material from a binary | If FairPlay-encrypted (`cryptid=1`), tool output is ciphertext ‚Äî the read is invalid, not a finding | binary is decrypted (`cryptid=0`) and the key is genuinely present and used |
| `arc4random` "weak RNG" | Fine for non-security jitter/sampling | output feeds a key, IV/nonce, token, or salt (use `SecRandomCopyBytes`) |
| Bundled OpenSSL/BoringSSL with DES/RC4 symbols | Algorithm present in the lib but never invoked by the app | the app actually calls the broken algorithm on real data |
| Software-only key (no Secure Enclave) | Acceptable under **L1** for low-value data | expected & absent under **L2/R**, or for high-value/long-lived keys |
| "Hardcoded key" string | A public key, OAuth `client_id`, cert pin, or non-secret constant | a private/symmetric key or secret that actually decrypts/authenticates data |

When you dismiss a candidate, record it (and the reason) in the report's "candidates not promoted" appendix.
