# iOS — MASVS-STORAGE (static)

Static review of how an iOS artifact (`.ipa` / `.app` bundle / Mach-O) stores sensitive data at rest and where it leaks. Candidate generators only — promote to a finding only after the anti-false-positive gate in SKILL.md.

Anchor each finding to the exact control in SKILL.md (read it); verify MASWE/MASTG IDs against the live guide.

> **FairPlay caveat (read first).** Against an App Store `.ipa` the `__TEXT` executable region is FairPlay-encrypted (`cryptid=1`), so `otool`/`nm`/`class-dump`/Ghidra read **ciphertext** — every code-level conclusion (API-usage greps below) is invalid until the binary is decrypted (`MASTG-TECH-0054` "Obtaining and Extracting Apps"; verify `cryptid=0` with `rabin2 -I`). Resource files (`Info.plist`, `.plist`, bundled DBs) are **not** FairPlay-encrypted and stay readable. Say which build you have.

## Group controls (MASVS v2.0.0, verbatim)
- **MASVS-STORAGE-1** — The app securely stores sensitive data.
- **MASVS-STORAGE-2** — The app prevents leakage of sensitive data.

First decide **what is sensitive in THIS app** (credentials, tokens, PII, keys, financial/health data). Then check where it lands and how it is protected. Entry techniques: `MASTG-TECH-0066` (Static Analysis on iOS), `MASTG-TECH-0070` (Extracting Information from the Application Binary), `MASTG-TECH-0078` (Automated Static Analysis, iOS); `MASTG-TECH-0153`/`-0154` for `Info.plist`, `MASTG-TECH-0111` for entitlements.

## Checks

- **Unencrypted sensitive data in private storage — API surface.** Grep the (decrypted) binary / source for plaintext-write APIs into the sandbox: `UserDefaults`/`NSUserDefaults` `set(_:forKey:)`; `NSKeyedArchiver` / `Codable` / `PropertyListSerialization` writing `.plist`; Core Data without `NSPersistentStoreFileProtectionKey` (or SQLCipher); raw `sqlite3_open`; `Data`/`String.write(to:)`, `FileManager.createFile`, `NSData writeToFile`. The fix pattern is encrypt-before-write with a Keychain-held key. → **MASVS-STORAGE-1**, MASWE-0006 (Sensitive Data Stored Unencrypted in Private Storage Locations), `MASTG-TEST-0300` (References to APIs for Storing Unencrypted Data in Private Storage — static). Runtime confirmation: `MASTG-TEST-0301`/`-0302`. Legacy umbrella `MASTG-TEST-0052`.
- **Data Protection / NSFileProtection class on files at rest.** Files written without `NSFileProtectionComplete` / `...CompleteUnlessOpen` are readable while the device is locked. Statically look for the protection level set on writes (`Data.write(options: .completeFileProtection)`, `FileManager` `setAttributes` `.protectionKey`, `open()` `O_*` flags) and for the `com.apple.developer.default-data-protection` entitlement; the container default is only `CompleteUntilFirstUserAuthentication`, so absence = weak. → **MASVS-STORAGE-1**, MASWE-0006, `MASTG-TEST-0300` (static API-reference test; runtime sibling `MASTG-TEST-0299` "Data Protection Classes for Files in Private Storage").
- **Keychain accessibility class.** Sensitive data in the Keychain (correct) but with a weak accessibility attribute: `kSecAttrAccessibleAlways`/`...AlwaysThisDeviceOnly` (deprecated, always readable), `kSecAttrAccessibleAfterFirstUnlock(ThisDeviceOnly)`; and omission of any `*ThisDeviceOnly` variant means the item **rides iCloud Keychain/backup to attacker hardware**. Baseline for high-value secrets is `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`. Grep the `SecItemAdd`/`SecItemUpdate` attribute dict and `SecAccessControlCreateWithFlags`. (Primary anchor is key management — **MASVS-CRYPTO-2** / MASWE-0018 / `MASTG-TEST-0300`; included here because it is the central at-rest control. Anchor a real finding by statement and cross-reference CRYPTO.)
- **Secure screen-lock not verified / `WhenPasscodeSet` not used.** For high-value secrets, tie data to the device passcode: `kSecAttrAccessibleWhenPasscodeSet(ThisDeviceOnly)` encrypts the Keychain item only while a passcode is set (and drops it if the passcode is removed), and the app should verify a secure lock exists via `LAContext.canEvaluatePolicy(.deviceOwnerAuthentication)` before relying on local auth/Keychain protection. Absence → secrets stay decryptable on a no-passcode device. → **MASVS-RESILIENCE-1** · MASWE-0008 (Missing Device Secure Lock Verification Implementation) · `MASTG-TEST-0248` (References to APIs for Detecting Secure Screen Lock — static, L2); **STORAGE-1 data-protection cross-cut** (anchor by statement; the protection-class angle is the STORAGE dimension).
- **Plaintext stores you can read without decryption.** Even against a FairPlay-encrypted `.ipa`, bundled/written resources are readable: dump `Library/Preferences/*.plist`, app-group containers (`group.*`), `*.sqlite`/Core Data stores, cached files, and any `Documents`/`Library` artifact for secrets in cleartext. Confirm sensitivity, then map to the write API above. → **MASVS-STORAGE-1**, MASWE-0006, `MASTG-TEST-0300` (runtime on-disk confirmation `MASTG-TEST-0302`).
- **Sensitive data in shared / no-interaction storage.** Data pushed to locations reachable without user interaction (shared app-group containers consumed by extensions, files surfaced to the Files app / iCloud Drive, `UIDocumentInteractionController` exports) stored unencrypted. → **MASVS-STORAGE-2**, MASWE-0007 (Sensitive Data Stored Unencrypted in Shared Storage Requiring No User Interaction), `MASTG-TEST-0303` (References to APIs for Storing Unencrypted Data in Shared Storage — static).
- **Logging of sensitive data.** `NSLog`, `os_log`/`Logger` (esp. `.public` interpolation or default formatting), `print`/`debugPrint`, `fprintf(stderr,…)` emitting tokens/PII/keys; logs persist in the unified log and are reachable on a paired/backed-up device. Grep these sinks and trace their arguments. → **MASVS-STORAGE-2**, MASWE-0001 (Insertion of Sensitive Data into Logs). No confirmed static atomic test — cite the control + "Sensitive data exposure via logging (static), ID to confirm"; runtime atomic tests are `MASTG-TEST-0296`/`-0297`, legacy umbrella `MASTG-TEST-0053`.
- **Backup exclusion for sensitive files.** Sensitive files in the sandbox not excluded from iCloud/iTunes/Finder backup, i.e. missing `URLResourceValues.isExcludedFromBackup = true` / `NSURLIsExcludedFromBackupKey` (and not relying on a `Caches`-dir placement). → **MASVS-STORAGE-2**, MASWE-0004 (Sensitive Data Not Excluded From Backup), `MASTG-TEST-0215` (Sensitive Data Not Marked For Backup Exclusion — static + dynamic). Runtime sibling `MASTG-TEST-0298`. Separately, **MASWE-0003** (Backup Unencrypted) covers the backup being unencrypted at all.
- **Third-party / SDK data sharing to disk.** Analytics, crash, or ad SDKs persisting sensitive payloads to their own caches/plists, or copying app data into shared containers. Confirm what the SDK writes and where. → **MASVS-STORAGE-2**, MASWE: unmapped (cross-ref PRIVACY-1 for the sharing angle); legacy `MASTG-TEST-0054` (third-party data sharing). Cite control + descriptive name + "(ID to confirm)" for an atomic test.
- **Keyboard cache / pasteboard / screenshots (leakage, cross-referenced to PLATFORM-3).** Text fields not marked secure cache keystrokes; the general `UIPasteboard` is system-wide and persistent; backgrounding snapshots can capture secrets. These are UI-exposure weaknesses (handled in PLATFORM-3) but are also STORAGE-2 *leakage at rest*. Keyboard-cache static check: missing `textContentType`/`isSecureTextEntry`/`autocorrectionType = .no` on sensitive fields → `MASTG-TEST-0313` (References to APIs for Preventing Keyboard Caching of Text Fields — static), which maps to **MASWE-0053** (Sensitive Data Leaked via the User Interface → anchor **MASVS-PLATFORM-3**, not STORAGE). Legacy `MASTG-TEST-0055` (keyboard cache). Keep the finding under PLATFORM-3; note the STORAGE-2 leakage dimension rather than double-reporting.
- **Memory residue (candidate only).** Swift `String`/`Data`/`NSString` holding keys or passwords that are never zeroed remain recoverable from a memory image. Static can only flag the candidate — mark **Needs dynamic verification** (heap dump). → **MASVS-STORAGE-2**, MASWE: unmapped; legacy `MASTG-TEST-0060`. One line, not a section.

> Note on legacy iOS storage IDs: the narrative umbrellas `MASTG-TEST-0052` (local storage), `-0053` (logs), `-0054` (third-party sharing), `-0055` (keyboard cache), `-0058`/`-0215` (backups) are being split into the atomic `0296`–`0303` / `0313` series above (static and runtime variants live side by side). Prefer the atomic static test and note the legacy umbrella.

## Classic false positives
Consult before promoting a candidate. Each is **usually NOT a finding** absent extra evidence:

| Candidate | Why it's usually a false positive | When it *is* real |
|---|---|---|
| Strings/symbols "leaking" secrets from the binary | If the binary is **FairPlay-encrypted** (`cryptid=1`), tool output is ciphertext — the read is invalid, not a finding | binary is decrypted (`cryptid=0`) and the secret is genuinely present |
| Data in `UserDefaults`/`.plist` flagged "unencrypted" | Non-sensitive config/UI state (flags, last-screen, feature toggles) doesn't need encryption | the value is a credential/token/PII/key, or gates auth/authz |
| File "without Complete protection" | Item is non-sensitive, or sits in `Caches`/`tmp` and is regenerable | a secret is written with `None`/`UntilFirstUserAuthentication` and is readable while locked |
| Keychain item "weak accessibility" | `AfterFirstUnlock` is intentional for a background-refresh token that must survive lock | high-value secret uses `Always`/no `*ThisDeviceOnly` and rides iCloud backup off-device |
| Not excluded from backup | The file is non-sensitive or already in a non-backed-up dir (`Caches`/`tmp`) | a credential/key/PII file in `Documents`/`Library` is eligible for iCloud/iTunes backup |
| `NSLog`/`os_log`/`print` "logs sensitive data" | Logs a non-sensitive value, or the call is compiled out in release (`#if DEBUG`) | a secret/PII reaches the unified log in the **release** build |
| MD5/SHA-1 seen near storage | Used as a cache key / file-name hash | used to "protect" stored sensitive data or as an integrity check on trusted-after-read data |

When you dismiss a candidate, record it (and the reason) in the report's "candidates not promoted" appendix.
